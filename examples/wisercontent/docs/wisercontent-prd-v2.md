# WiserContent Platform: Product Requirements v1.0

**Owner:** Bill / Feba Capital
**Stack:** PHP Yii2 (Advanced), MySQL, Dragonfly (Yii2 Queue/Redis), Langfuse
**Architecture:** API-first, multi-tenant, single-database
**Status:** v1.0 (Master PRD)

---

## 1. Overview

WiserContent is an API-first, multi-tenant content generation and publishing platform for Feba Capital's portfolio companies. Each tenant (portfolio company) operates as a **virtual content department**: an autonomous content factory with its own company profile, AI writers, brand/quality revisers, a brand strategist, publishing webhooks, and content cadence schedule.

Content flows through a component-level AI pipeline. Each piece of content is decomposed into individual components (title, SEO title, description, SEO description, body, images), and each is generated by the optimal AI model for that task. An automated scheduling system ("agent consortium") continuously evaluates approved briefs and assigns them to publishing slots based on cadence, category balance, and keyword freshness.

### 1.1 Goals
- Centralize blog content production across multiple portfolio companies.
- Enforce per-tenant isolation of users, data, keys, and prompts.
- Support competing AI-generated drafts with human approval.
- Publish via generic webhook to each company's custom platform.
- **Automate the full brief-to-publish pipeline:** from brief approval through component-level generation, revision, scheduling, and publishing, with minimal human intervention.
- **Enable per-component AI model selection:** use the best model for each content component (text vs. image).
- **Generate images** as a first-class content component alongside text.
- **Provide a virtual content department:** company profile, brand strategist (AI brief generation), typed and chained revisers (brand compliance, quality, SEO), all informed by company context.

### 1.2 Platform Evolution

| Area | MVP (v0.x) | v1.0 | v1.1 (VCD) |
|---|---|---|---|
| Queue backend | File-based | Dragonfly (Redis-compatible) with channel routing | - |
| Content generation | Monolithic `GenerateDraftJob` | Component-level jobs (title, SEO, body, image) | Company profile context in all prompts |
| AI models | Single model per writer | Per-component model config per writer | - |
| Image generation | Out of scope | Gemini 2.5 via `AiProviderService` | - |
| Scheduling | Manual schedule-per-item | Automated cadence with slot-based scheduling | - |
| Brief creation | Manual + AlsoAsked + keyword research | Agent consortium with category/keyword balancing | AI Brand Strategist (brief generation from company profile) |
| AlsoAsked data | Flat question list | Hierarchical with level/depth tracking | - |
| Revision | Single default reviser | Single default reviser | Typed & chained revisers (brand → quality → SEO) |
| Tenant identity | name + slug | brand_name + niche_description + keywords | Full company profile (products, audiences, brand voice, competitors, guidelines) |

### 1.3 Non-Goals
- First-party admin UI (API only).
- Analytics on published performance.
- A/B testing of prompts.
- Built-in CMS / WYSIWYG.
- Fallback chains across AI providers (fail loudly).
- Prompt caching (always pull fresh from Langfuse).
- Outbound webhooks for lifecycle events.
- Multi-provider fallback chains.
- Spend/budget caps.

## 2. Tenancy & Users

### 2.1 Isolation Model
- **Single MySQL database**, row-level scoping via `tenant_id BIGINT UNSIGNED` on every tenant-owned table.
- **All IDs are `BIGINT UNSIGNED`.** No string IDs, no UUIDs.
- Global middleware enforces tenant scope on every authenticated request; queries without a `tenant_id` filter are rejected at the ActiveRecord layer.

### 2.2 Users & Roles

**Platform-level role (above all tenants):**

| Role | Permissions |
|---|---|
| `PlatformAdmin` | Feba-level super-admin. Creates/manages tenants, views cross-tenant data, manages platform-level config (AlsoAsked key, global Langfuse project, content type enum). Not scoped to a single tenant. Can impersonate any tenant. Can delete tenants (soft-delete to inactive). |

**Tenant-level roles (per tenant membership):**

| Role | Permissions |
|---|---|
| `Owner` | Full tenant control incl. user management (add/remove), AI keys, Langfuse config, publish targets, cadence schedule, company profile. Can approve/reject content. Can delete users from tenant. |
| `Admin` | Manage categories, writers, revisers, schedule, publish targets, AI keys, cadence config, company profile. Can approve/reject content. Can trigger AI brief generation. |
| `Editor` | Create/edit briefs, trigger generation, edit drafts, **approve/reject content** (including their own; no four-eyes rule), reschedule/cancel scheduled items. |
| `Viewer` | Read-only |

- **No separate Approver/Reviser role.** Editor, Admin, and Owner can all approve.
- **No four-eyes rule:** an Editor can approve content they authored or triggered.
- **Tenant user isolation:** a user's data within one tenant is never visible to users of another tenant (except to `PlatformAdmin`).

### 2.3 Cross-Tenant Users
- A single user account may belong to multiple tenants via a `user_tenant` join table carrying a per-membership role.
- JWT access tokens embed an **active `tenant_id`**; switching tenants issues a new token scoped to the new tenant.
- All authorization checks use the active tenant from the token, never the user's full tenant list.
- **`PlatformAdmin`** is a separate platform-level flag on the user record (not per-tenant). PlatformAdmins can impersonate any tenant context for support and cross-tenant reporting.
- **Refresh tokens** store `tenant_id` and `role`; refreshing restores the tenant context without re-login.

### 2.4 Invitation, Password Reset & MFA
- **User invitation:** three paths supported:
  1. **Email invite:** signed time-limited token; invitee sets their own password via `POST /v1/auth/accept-invitation`.
  2. **Direct admin creation:** admin creates user with `mode=direct`; system generates a 16-char temporary password, sets `must_reset_password=true`, returns the temporary password and a reset token. User must reset password before first login (login returns 403 with `must_reset_password: true` hint).
  3. **Password-provided:** admin supplies the password directly at creation time.
- **Password reset:** standard forgot-password flow with SHA-256 hashed, time-limited, single-use token. Clears `must_reset_password` flag if set.
- **MFA:** TOTP (RFC 6238). Optional per user; enforceable per tenant by `Owner`.

### 2.5 Tenant Entity & Keywords
Each tenant represents a **brand entity** defined by two identity fields:
- **`brand_name`**: the company or brand name (e.g., "SleepWell Co")
- **`niche_description`**: the market niche or vertical (e.g., "premium sleep products and mattress reviews")

Tenants maintain a set of **keywords** in a separate `tenant_keywords` table:
- **Primary keywords:** core terms that define the tenant's domain. Automatically combined with AlsoAsked search queries to produce richer, more targeted question results. For example, if a tenant has primary keyword "sleep health" and a user queries "best mattress", the system sends terms `["best mattress", "best mattress sleep health"]` to the AlsoAsked API.
- **Secondary keywords:** supporting terms stored for reference and future AI-driven content strategies.

Keywords are managed via Owner/Admin CRUD endpoints (`/v1/tenant-keywords`). Each keyword is unique per tenant.

## 3. Virtual Content Department

### 3.1 Overview
Each tenant operates a **virtual content department**: an org chart of AI roles that collaborates to produce brand-aligned content:

```
               Company Profile (context for all agents)
                        |
                Brand Strategist
              (generates brief ideas)
                        |
           +-----------+-----------+
           |           |           |
      Writer A     Writer B     Writer C
     (technical)  (casual)    (SEO specialist)
           |           |           |
           +-----------+-----------+
                        |
              Revision Chain (ordered)
           +---> Brand Reviser (compliance)
           +---> Quality Reviser (grammar/structure)
           +---> SEO Reviser (keyword optimization)
                        |
              Pending Approval (human)
```

### 3.2 Company Profile (Tenant Profile)
Each tenant has an optional rich company profile (`tenant_profiles` table) that provides context to all AI agents:

| Field | Type | Description |
|---|---|---|
| `company_description` | TEXT | What the company does |
| `mission_statement` | TEXT | Company mission/values |
| `products` | JSON | Array of `{name, description, url}` |
| `target_audiences` | JSON | Array of `{segment, description, pain_points}` |
| `brand_voice` | JSON | `{tone, personality, dos, donts}` |
| `competitors` | JSON | Array of `{name, url, notes}`, used for exclusion in prompts |
| `content_guidelines` | TEXT | Specific content rules and standards |
| `industry` | VARCHAR(255) | Industry vertical |

- One profile per tenant (unique constraint on `tenant_id`).
- Managed via `GET/PUT /v1/tenant-profile` (Admin+, upsert pattern).
- Profile data is injected into all AI prompts:
  - **Writer prompts:** company_description, mission_statement, products, target_audiences, brand_voice, competitors, content_guidelines, industry passed as prompt variables to all component jobs.
  - **Reviser prompts:** brand_voice, competitors, content_guidelines passed to revision jobs for brand-aware revision.
  - **Brand Strategist prompts:** full profile used to generate brand-aligned brief ideas.
- Profile is **optional**. All pipeline functions work without one, using only tenant-level brand_name and niche_description.

### 3.3 Brand Strategist (AI Brief Generation)
The Brand Strategist is a synchronous AI service (`BriefGenerationService`) that generates content brief ideas from a topic, informed by the tenant's company profile.

**Endpoint:** `POST /v1/briefs/generate`
**Required role:** Admin+
**Request body:**
```json
{
  "topic": "sleep hygiene tips",
  "count": 5,
  "ai_provider_id": 1  // optional, defaults to first available
}
```
**Response:** 201 with created briefs (source = `ai_strategist`)

**How it works:**
1. Loads the tenant's company profile (description, products, audiences, brand voice, competitors, guidelines, industry).
2. Loads the tenant's primary keywords.
3. Builds a Brand Strategist prompt with all company context + the requested topic.
4. Calls the tenant's AI provider directly, not via Langfuse; it uses an inline prompt.
5. Parses the AI response (JSON array of brief objects).
6. Creates `content_briefs` records with `source = 'ai_strategist'`.

**Brief sources (complete list):**
- `manual`: created directly by tenant users
- `alsoasked`: created from AlsoAsked API questions
- `keyword_research`: created from AI keyword expansion
- `ai_strategist`: created by the Brand Strategist from company profile context

### 3.4 Reviser Types & Ordering
Revisers have two additional fields beyond the MVP spec:

| Field | Type | Description |
|---|---|---|
| `revision_type` | ENUM | `brand_compliance`, `quality`, `seo`, `custom` (default: `custom`) |
| `revision_order` | INT UNSIGNED | Execution order in the revision chain (default: 0) |

**Revision types:**
- **`brand_compliance`**: checks tone, ensures no competitor promotion, verifies brand alignment against company profile
- **`quality`**: grammar, structure, readability, factual consistency
- **`seo`**: keyword optimization, meta tag quality, heading structure
- **`custom`**: any tenant-defined revision purpose

### 3.5 Chained Revision Pipeline
After draft assembly, the system runs an **ordered chain of revisers** instead of a single default reviser:

1. `AssembleContentJob` queries all **active** revisers for the tenant, ordered by `revision_order ASC`.
2. If no revisers found → transition directly to `pending_approval`.
3. If revisers found → dispatch `RunReviserJob` with the first reviser and the full chain.
4. After each revision:
   - If more revisers remain in the chain → dispatch `RunReviserJob` for the next reviser.
   - If last in chain → transition to `pending_approval`.
5. Each reviser in the chain creates its own revision round.

**State transitions for chaining:**
- `written → revised` (first reviser)
- `revised → revised` (subsequent revisers in chain)
- `revised → pending_approval` (after final reviser)

**Company context in reviser prompts:**
- All revisers receive `brand_voice`, `competitors`, and `content_guidelines` from the tenant profile.
- Brand compliance revisers use competitor names for explicit exclusion checks.
- Quality revisers receive content_guidelines for style conformance.

### 3.6 Company Profile in Generation Pipeline
The tenant's company profile is wired into the content generation pipeline at two levels:

**Component generation (`BaseComponentJob.buildVariables()`):**
- `{{company_description}}`: from profile
- `{{mission_statement}}`: from profile
- `{{products}}`: comma-separated product names
- `{{target_audiences}}`: comma-separated audience segments
- `{{brand_voice}}`: full JSON brand voice object
- `{{competitors}}`: comma-separated competitor names (for exclusion)
- `{{content_guidelines}}`: from profile
- `{{industry}}`: from profile

**Revision (`RunReviserJob`):**
- `{{brand_voice}}`: for tone checking
- `{{competitors}}`: for competitor mention detection
- `{{content_guidelines}}`: for compliance checking

**Fallback prompts (`BaseComponentJob.buildFallbackPrompt()`):**
When no Langfuse prompt is found, the fallback prompt template includes company context, industry, content_guidelines, and competitor exclusion instructions.

## 4. Infrastructure

### 4.1 Docker Services

```yaml
services:
  api:          # PHP 8.2 web (Traefik at api.wisercontent.test)
  console:      # PHP 8.2 CLI (migrations, queue workers, scheduler)
  dragonfly:    # Redis-compatible in-memory store (queue backend)
```

- **Dragonfly** replaces the file-based queue. Runs as a Docker service on port 6379 within the `dev-network`.
- No separate MySQL container. The database is external (`51.222.82.178:33060`).

### 4.2 Queue Backend
- **Driver:** `yii\queue\redis\Queue` pointing to Dragonfly.
- **Channels** route jobs to dedicated workers for isolation and prioritization:

| Channel | Jobs | Priority |
|---|---|---|
| `title` | GenerateTitleJob | Normal |
| `seo` | GenerateSeoTitleJob, GenerateDescriptionJob, GenerateSeoDescriptionJob | Normal |
| `body` | GenerateBodyJob | Normal |
| `image` | GenerateImageJob | Low |
| `revision` | RunReviserJob | Normal |
| `publish` | PublishContentJob | High |
| `scheduling` | ScheduleEvaluationJob | Normal |
| `research` | AlsoAsked/keyword research jobs | Low |
| `orchestration` | GenerateContentJob (orchestrator) | High |
| `default` | GenerateDraftJob (legacy), catch-all | Normal |

- **QueueRouter component** resolves channel-specific queue components (`queue-title`, `queue-seo`, etc.) when configured, falling back to the default queue.
- All queue dispatch sites route through `QueueRouter`.

### 4.3 Worker Topology
- At minimum, **one worker process per channel group** in production.
- Dev environment: a single worker consuming all channels is acceptable.
- Workers run via `php yii queue/listen --channel=<channel>` under supervisord or systemd.

### 4.4 Database
- MySQL 8+, InnoDB, utf8mb4_unicode_ci.
- All FKs enforced. `tenant_id` indexed on every tenant-owned table and included in composite indexes for hot queries.

### 4.5 Secrets
- App-level AES-256 master key from environment (not in repo).
- Encrypted columns: AI provider keys, Langfuse secret keys, webhook auth headers.

### 4.6 Observability
- Langfuse for AI traces (per-component trace IDs).
- Application logs to **stderr** (`php://stderr`), which is container-compatible.
- Health endpoint `GET /v1/health` returning DB latency, queue reachability (Redis PING), and Langfuse connectivity.

  - `tenant_id BIGINT UNSIGNED` (FK -> tenants.id CASCADE)

  - `content_item_id BIGINT UNSIGNED NULL` (FK -> content_items.id SET NULL)

### 5.5 AlsoAsked Keyword Tracking

  - `tenant_keyword_id BIGINT UNSIGNED` (FK -> tenant_keywords.id CASCADE)

  - `question VARCHAR(512)`

  - `fetched_at INT`

### 5.6 Encryption
AES-256 with app-level key, not sealed box. No per-tenant DEK/KEK hierarchy.

## 6. Content Generation Pipeline

### 6.1 Component-Based Architecture
Each content piece is decomposed into six components, each generated independently with the optimal AI model.

**Component types:**

| Component | Channel | Typical Model | Required |
|---|---|---|---|
| `title` | `title` | GPT-5.4-mini / Claude Sonnet 4.6 | Yes |
| `seo_title` | `seo` | GPT-5.4-mini / Claude Sonnet 4.6 | Yes |
| `description` | `seo` | GPT-5.4-mini / Claude Sonnet 4.6 | Yes |
| `seo_description` | `seo` | GPT-5.4-mini / Claude Sonnet 4.6 | Yes |
| `body` | `body` | GPT-5.4-mini / Claude Sonnet 4.6 | Yes |
| `image` | `image` | Gemini 2.5 | No (optional) |

### 6.2 Pipeline Flow

```
Brief Approved
  -> GenerateContentJob (orchestrator, channel: orchestration)
    -> Creates content_item (status: generating)
    -> Creates content_components (status: pending) for each type
    -> Dispatches component jobs in parallel:
        GenerateTitleJob         -> channel: title
        GenerateSeoTitleJob      -> channel: seo
        GenerateDescriptionJob   -> channel: seo
        GenerateSeoDescriptionJob -> channel: seo
        GenerateBodyJob          -> channel: body
        GenerateImageJob         -> channel: image

Each component job:
  1. Reads writer_model_configs for component_type -> resolves AI provider + model
  2. Falls back to writer.model + writer.ai_provider_id if no component config exists
  3. Fetches Langfuse prompt by label: {writer.langfuse_prompt_label}-{component_type}
  4. Interpolates prompt variables (brief data, personality metadata, company profile context)
  5. Calls AiProviderService
  6. Saves result to content_components
  7. Creates Langfuse trace with component-level metadata

Assembly (triggered when required components complete):
  -> AssembleContentJob assembles content_components into a content_draft
  -> Transitions content_item to "written"
  -> Auto-queues chained revision pipeline (RunReviserJob with reviser chain)
```

### 6.3 Failure Handling
- **Required components** (title, seo_title, description, seo_description, body): if any fails after retries, the content item enters `generation_failed`. Human must retry.
- **Optional components** (image): failure does not block the draft. The draft is assembled without the image.
- **Competing drafts:** up to 2 writers may generate drafts for the same brief. If one writer's set fails entirely and the other succeeds, proceed with the winner.
- **Retry policy:** 3 retries with exponential backoff (30s, 120s, 480s) via explicit delayed re-queuing. Retries are not managed by Yii2's built-in retry mechanism; jobs catch exceptions and re-queue themselves with incremented `attemptNumber`.

### 6.4 Assembly Logic
When all required components for a writer are `completed`:
1. Create a `content_draft` row with assembled body.
2. Store `component_refs` JSON with the IDs of all components used.
3. Increment `articles_generated` usage counter only on first draft assembly per content item, so competing drafts do not double-count.
4. If this is the last (or only) writer completing, transition the content item to `written`.
5. Auto-queue revision chain (§3.5).

### 6.5 Content Refresh / Revamp
- A published or archived content item may be **refreshed** to produce an updated version via `POST /v1/content/{id}/refresh` (Admin+).
- This creates a new `content_items` row with `parent_content_id` pointing to the original, preserving lineage. The original is not modified.
- The refresh re-runs the full component-level pipeline.
- Only items in `published` or `archived` status can be refreshed (returns 409 for other statuses).

## 7. Writers, Revisers & Model Configuration

### 7.1 Writer Structure
```
Writer = {
  tenant_id, name, ai_provider_id, model (default),
  langfuse_prompt_label, personality_metadata (JSON),
  category_assignments (JSON), content_type_assignments (JSON),
  is_default (bool), status
}
```

### 7.2 Per-Component Model Config
Each writer can have **per-component model overrides** via `writer_model_configs`:

```
writer_model_configs:
  writer_id: 42, component_type: "body",  ai_provider_id: 2, model: "gpt-5.4-mini"
  writer_id: 42, component_type: "title", ai_provider_id: 1, model: "claude-sonnet-4-6"
  writer_id: 42, component_type: "image", ai_provider_id: 3, model: "gemini-2.5-flash"
```

**Resolution order:**
1. Check `writer_model_configs` for the specific `(writer_id, component_type)`.
2. Fall back to the writer's default `ai_provider_id` + `model`.

### 7.3 Auto-Assignment
- **WriterAssignmentService** matches brief's `category_id` + `content_type` against writers' `category_assignments` and `content_type_assignments` JSON.
- If 2+ match, pick top 2 by priority; if 1 matches, generate 1 draft; if 0 match, fall back to the tenant's `is_default` writer.
- Manual Editor override always available via `writer_ids` parameter on generate endpoints.

### 7.4 Revisers
- Same structure as writers with additional `revision_type` and `revision_order` fields (§3.4).
- Auto-runs as a chained pipeline after writer completion (§3.5).
- Multiple revision rounds supported: each reviser in the chain creates its own round.

### 7.5 Prompt Labels
Langfuse prompts are organized per-component:
- Writer: `{langfuse_prompt_label}-title`, `{langfuse_prompt_label}-seo-title`, `{langfuse_prompt_label}-description`, `{langfuse_prompt_label}-seo-description`, `{langfuse_prompt_label}-body`, `{langfuse_prompt_label}-image`
- Reviser: `{langfuse_prompt_label}` (operates on assembled draft, not components)
- Prompts pulled **fresh at generation time** by label. **No caching.**

### 7.6 Personality & Brand Context
Prompt variables available to all component prompts:
- `{{personality}}`: full JSON from `writers.personality_metadata`
- Individual personality keys spread as separate variables (e.g., `{{tone}}`, `{{audience}}`)
- `{{brand_name}}`: from `tenants.brand_name`
- `{{niche_description}}`: from `tenants.niche_description`
- `{{primary_keywords}}`: from `tenant_keywords` where type = primary
- `{{category}}`: from the brief's category
- `{{keywords}}`: from the brief's keyword list
- `{{title}}`: overridden with generated title component content for body/image prompts
- `{{description}}`: overridden with generated description for body/image prompts
- `{{company_description}}`, `{{mission_statement}}`, `{{products}}`, `{{target_audiences}}`, `{{brand_voice}}`, `{{competitors}}`, `{{content_guidelines}}`, `{{industry}}`: from tenant profile (§3.6)

### 7.7 Writer/Reviser Config Versioning
- **No snapshotting.** Generation always uses the **latest production prompt** from Langfuse and the **current writer/reviser config** at execution time.

## 8. Content Scheduling & Cadence

### 8.1 Tenant Schedule Configuration
Each tenant configures a **content cadence** defining how often and when content should be published:

```json
{
  "posts_per_week": 3,
  "publish_days": ["monday", "wednesday", "friday"],
  "publish_window_start": "09:00:00",
  "publish_window_end": "17:00:00",
  "auto_generate_on_approval": true,
  "auto_schedule_on_approval": true
}
```

- Times are in the **tenant's IANA timezone**; stored/compared in UTC internally.
- If no schedule is configured, the tenant operates in manual-only mode.

### 8.2 Slot Generation
A scheduler worker pre-generates `scheduled_slots` for each tenant:
- Generates slots **2 weeks ahead** based on `publish_days` and `publish_window`.
- Each slot has a specific `slot_date` and `slot_time_utc` (randomly distributed within the publish window for natural posting cadence).
- Slots start as `open` and are filled by the agent consortium or manual scheduling.
- Slot generation runs daily as console cron: `php yii slot-generator/run`.

### 8.3 Manual Scheduling
- Admins/Editors can manually schedule content to specific times via `POST /v1/content/{id}/schedule`.
- Manual scheduling creates or fills a slot. If the time falls outside the cadence window, the slot is created anyway (manual override).
- Manual scheduling takes priority over auto-scheduling.
- Reschedule via `PUT /v1/content/{id}/reschedule`, which updates schedule time and/or publish target (Editor+).

### 8.4 Scheduler Worker
- **`php yii scheduler/run`**: runs every minute via cron.
- Polls `scheduled_slots` where `status = 'filled'` and `slot_time_utc <= NOW()`.
- Validates content item is in `scheduled` status before dispatching.
- Dispatches `PublishContentJob` for each due slot.
- Updates slot status to `published` on success.

## 9. Agent Consortium (Automated Content Orchestration)

### 9.1 Overview
The agent consortium is a per-tenant automated system that orchestrates the full brief-to-schedule pipeline. It runs as `ScheduleEvaluationJob` (channel: `scheduling`), triggered on brief approval or by periodic cron (every 15 minutes via `php yii schedule-evaluator/run`).

### 9.2 Two-Phase Execution

**Phase 1: Auto-schedule approved content**
- Finds approved content items without slots.
- Assigns them to open `scheduled_slots`.
- Sets `publish_target_id` from tenant's first publish target.

**Phase 2: Create content from approved briefs**
- Finds approved briefs without content items.
- Creates content items and dispatches `GenerateContentJob`.
- Assigns all matched writers (up to 2 competing drafts).

### 9.3 Brief Selection Criteria
When multiple approved briefs are available and slots are open:

| Criterion | Weight | Description |
|---|---|---|
| Category distribution | High | Spread content across all tenant categories; avoid consecutive posts in same category |
| Keyword freshness | High | Prefer keywords not covered in recent publications; use `alsoasked_keywords.level` for deeper questions |
| Content type variety | Medium | Mix content types across the publishing calendar |
| Brief age | Medium | Older approved briefs get priority to prevent stale queues |
| AlsoAsked depth | Medium | Deeper questions (higher `level`) indicate more specific, less competitive content |

### 9.4 Audit Trail
Every consortium decision is audit-logged with `ACTION_CONSORTIUM_DECISION`, recording the brief, slot, category, and writer details assigned.

## 10. Content Workflow

### 10.1 Brief Lifecycle
```
Brief Draft -> Brief Approved -> Generation Triggered (auto or manual)
```
- **Briefs require approval** before generation runs (generation costs money).
- **Bulk pre-approval:** Admin/Owner may pre-approve an entire month's schedule in one batch.
- **Auto-generation:** on brief approval (if `auto_generate_on_approval = true`), the system dispatches `GenerateContentJob` and `ScheduleEvaluationJob`. Manual trigger also available.
- **Brief approval** creates explicit `ACTION_APPROVE` audit log entry.
- **Brief rejection** creates explicit `ACTION_REJECT` audit log entry with required comment.

### 10.2 Content State Machine

```
Draft
  -> Generating          (component jobs dispatched, running in parallel)
  -> Written             (all required components completed, draft assembled)
  -> Revised             (reviser auto-runs; may chain to next reviser)
  -> Pending Approval    (human may edit inline here)
  -> Approved
  -> Scheduled           (assigned to a slot or manually scheduled)
  -> Published
  -> Archived            (90 days post-publish, no deletion)

Error states:
  -> Generation Failed   (required component failed after all retries)
  -> Publish Failed      (webhook failed after all retries)
```

**11 states total.** Every state transition is audit-logged with from/to status, user ID, and optional comment.

### 10.3 Transition Rules
- **Draft -> Generating:** on `GenerateContentJob` dispatch. Component jobs run in parallel.
- **Generating -> Written:** when all required components for at least one writer complete and the draft is assembled.
- **Written -> Revised:** **automatic.** First reviser in chain runs immediately after draft assembly.
- **Revised -> Revised:** allowed (next reviser in chain runs).
- **Revised -> Written:** allowed (re-run writer; multiple revision rounds supported).
- **Revised -> Pending Approval:** automatic when last reviser in chain completes, or if no revisers exist.
- **Pending Approval:** Editor, Admin, or Owner may edit the selected draft body inline. **Only the latest edit is kept. No snapshots.**
- **Pending Approval -> Draft:** allowed for rejection to any earlier stage.
- **Approval:** Editor, Admin, or Owner can approve. **No four-eyes rule.**
- **Reject with comment:** approver may send back to any earlier stage with a **required** comment.
- **Approved -> Scheduled:** automatic via agent consortium (if enabled) or manual via schedule endpoint.
- **Scheduled -> Archived:** allowed for cancel/archive from scheduled state.
- **Reschedule/Cancel:** Editor, Admin, or Owner while item is `Scheduled`.
- **Published -> Archived:** automatic 90 days after publish via `php yii archive/run` (daily cron, batch of 100). **Status change only. No data is ever deleted.**
- **No unpublish/takedown flow.** Once a publish webhook succeeds, content is the receiving platform's responsibility.

### 10.4 Competing Drafts
- Up to **2 writers** may generate drafts for the same brief; **1 draft is also acceptable**.
- Each writer generates a full set of components independently.
- During `Pending Approval`, the approver picks the winning draft when two exist.
- **Failure:** if both writers' component sets fail -> `generation_failed`. If one succeeds -> proceed with the winner.

## 11. Image Generation

### 11.1 Overview
Image generation is a first-class content component, generated alongside text components as part of the content pipeline.

### 11.2 Provider
- **Gemini 2.5** via the Google AI provider in `AiProviderService`.
- `AiProviderService.image()` method for image generation requests (separate from `chat()`).
- Image prompts managed in Langfuse (label: `{writer.langfuse_prompt_label}-image`).

### 11.3 Storage
- Generated images stored as **base64-encoded data** in `content_components.content` for v1.0/v2.0.
- Future: migrate to object storage (S3/GCS) with URL references.
- Image metadata (dimensions, format, generation params) stored in `content_components.metadata`.

### 11.4 Behavior
- Image is an **optional component**. Failure does not block draft assembly.
- If image generation fails, the draft proceeds without an image.
- Image prompt receives all the same variables as text prompts plus `{{title}}` and `{{description}}` from completed text components.
- Retry: 3 retries with exponential backoff (30s, 120s, 480s), same as text components.

## 12. AlsoAsked Integration & Keyword Tracking

### 12.1 API Integration
- **Single platform-level API key** stored in application config (not per tenant).
- **Async API:** POST terms array -> job ID -> poll GET until success.
- Tenant primary keywords are auto-combined with queries (§2.5).

### 12.2 Hierarchical Question Storage
When AlsoAsked questions are fetched, they are stored in `alsoasked_keywords` with full hierarchy:
- `level 0`: the original search term/query (root entry)
- `level 1`: first-level "People Also Asked" questions
- `level 2`: sub-questions from level 1 answers
- `parent_id`: links to the parent question in the tree

This hierarchical data enables:
- **Depth-based brief prioritization:** deeper questions (level 2+) indicate more specific, less competitive topics.
- **Topic clustering:** related questions under the same parent form natural content clusters.
- **Coverage tracking:** `brief_id` on `alsoasked_keywords` tracks which questions have been turned into briefs.

Level-0 root entries (raw query terms) are filtered out before brief creation. Only actual PAA questions (level 1+) become briefs.

### 12.3 Usage Attribution
- AlsoAsked API usage is attributed to the calling tenant for monthly counters.
- No per-tenant AlsoAsked quota.

## 13. Content Briefs & Keyword Research

### 13.1 Brief Sources
1. **Manual entry** by tenant users (title, category, content type, keywords).
2. **AlsoAsked API:** tenant provides a seed query; platform fetches related questions, stores them with hierarchy, and creates briefs from level 1+ questions.
3. **Keyword research pre-brief:** global standard research prompt runs against the tenant's AI provider to expand a topic into a keyword set, which seeds a brief.

### 13.2 Brief Structure

### 13.4 Content Types (Global Fixed Enum)
Platform-wide enum, not tenant-configurable:
`how_to`, `listicle`, `explainer`, `news`, `opinion`, `case_study`, `interview`, `review`, `comparison`, `guide`.

## 16. Scheduling & Publishing

### 16.1 Publish Targets
- Generic webhook only. Each tenant configures one or more `publish_targets` with a webhook URL, optional auth header, and a per-tenant payload template.
- **One target per content item.** Multi-target fan-out not supported.
- Target selection at schedule time, typically defaulted from the tenant's first publish target.

### 16.2 Per-Tenant Payload Schema
Each `publish_target` stores a **JSON payload template** with placeholders:
- `{{title}}`: from title component
- `{{seo_title}}`: from seo_title component
- `{{description}}`: from description component
- `{{seo_description}}`: from seo_description component
- `{{body_html}}`: from body component
- `{{body_markdown}}`: from body component (markdown format)
- `{{image_base64}}`: from image component (if available)
- `{{category}}`, `{{keywords}}`, `{{content_id}}`, `{{published_at}}`

### 16.3 Timezone
- Each tenant has an IANA timezone; users schedule in local TZ, storage is **UTC**.

### 16.4 Retry
- On non-2xx, retry 3x with exponential backoff (30s, 120s, 480s), then mark item `publish_failed`.
- Response code + body persisted in `publish_attempts`.

### 16.5 Console Workers
| Worker | Schedule | Description |
|---|---|---|
| `php yii scheduler/run` | Every minute | Polls filled slots, dispatches PublishContentJob |
| `php yii schedule-evaluator/run` | Every 15 minutes | Evaluates briefs for open slots (agent consortium) |
| `php yii slot-generator/run` | Daily | Pre-generates 2 weeks of slots per tenant |
| `php yii archive/run` | Daily | Archives published content after 90 days (batch of 100) |

## 17. API Design

### 17.1 Conventions
- **URI versioning:** `/api/v1/...`
- **Auth:** JWT access tokens with embedded `user_id`, active `tenant_id`, and role. 15-min access tokens, 30-day rotating refresh tokens. Refresh tokens preserve tenant context.
- **Content type:** `application/json`.
- **Success envelope:** `{success: true, data: {...}, meta: {...}}`
- **Error envelope:** `{success: false, error: {code: string, message: string, details: object}}`
- **Pagination:** cursor-based, `?cursor=&limit=` with `next_cursor` in response.

### 17.2 Endpoint Groups
- `/auth`: login, refresh, switch-tenant, me, logout, forgot-password, reset-password, mfa/setup, mfa/verify, mfa/disable, accept-invitation, impersonate
- `/tenants`: CRUD (PlatformAdmin creates; soft-delete sets status to inactive)
- `/tenant-keywords`: CRUD (Owner/Admin), filterable by type
- `/tenant-profile`: get/set company profile (Admin+, upsert pattern)
- `/tenant-schedule`: get/set cadence config (Admin+, upsert pattern)
- `/users`: CRUD within tenant (Owner can delete users, cannot delete self)
- `/categories`: tenant CRUD (deletion blocked if referenced)
- `/writers`: tenant CRUD, category assignments, per-component model configs
- `/revisers`: tenant CRUD, revision_type, revision_order
- `/ai-keys`: tenant CRUD per provider
- `/langfuse-config`: tenant get/set
- `/briefs`: CRUD, manual create, alsoasked-create, keyword-research-create, approve, reject, batch-approve, **generate** (AI Brand Strategist)
- `/content`: list, get, generate, select-draft, edit, approve, reject, schedule, reschedule, publish-now, cancel, retry, refresh
- `/content/{id}/drafts`: list, get draft body
- `/content/{id}/revisions`: trigger reviser, history
- `/content/{id}/components`: list component status/content per content item
- `/publish-targets`: tenant CRUD
- `/scheduled-slots`: list upcoming slots (Admin+)
- `/alsoasked-keywords`: list fetched questions with depth (Admin+)
- `/usage`: monthly counters per tenant
- `/health`: unauthenticated health check (DB latency, queue PING, Langfuse reachability)
- `/audit-log`: list audit entries with action, entity_type, entity_id, user_id filters (Admin+, PlatformAdmin cross-tenant)

### 17.3 Async Operations
- Generation, revision, and publish are all queued. Endpoints return `202 Accepted` with a `job_ids` (array) or `job_id` (string|integer) reference; clients poll `/content/{id}` for status updates.
- Component-level status visible via `/content/{id}/components`.

## 18. Security & Audit

- OAuth2 + JWT, short-lived access tokens (15 min), refresh tokens (30 days, rotating, with tenant context).
- **MFA (TOTP)** supported, enforceable per tenant.
- **`must_reset_password`** flag for admin-created accounts: login returns 403 until password is reset.
- **No rate limiting.**
- Strict tenant scoping enforced in a base ActiveRecord trait.
- `PlatformAdmin` impersonation actions are always audit-logged.

### 18.1 Audit Log
All captured in `audit_log` indexed by `user_id` and `tenant_id`:
- Every content **state transition** (from/to, user, comment).
- Every **component generation** event (model, provider, tokens, cost).
- Every **config change** (writers, revisers, categories, publish targets, AI keys, Langfuse config, cadence schedule, **company profile**).
- Every **AI key rotation** (old key fingerprint only).
- Every **user action** on authentication (login, logout, password reset, MFA).
- Every **brief approval/rejection** (individual or batch, with explicit action types).
- Every **PlatformAdmin impersonation**.
- Every **agent consortium decision** (brief, slot, category, writer details).

Retention: **indefinite** (no automatic purge). Table is append-only.

## 19. Edge Cases & Known Constraints

### 19.1 Virtual Content Department
- **Empty company profile:** All pipeline functions work without a profile. Writer/reviser prompts simply omit company context variables. Brand Strategist generates generic briefs without company context.
- **No revisers configured:** Draft assembly transitions directly to `pending_approval`, skipping the revision chain entirely.
- **Single reviser:** Chain of length 1 behaves identically to the MVP single-reviser pattern.
- **Reviser chain failure:** If a reviser in the chain fails (AI error), the draft remains in `revised` state. The failed revision round is recorded. Human must manually re-trigger or skip.
- **Brand Strategist with no AI key:** Returns error. Tenant must have at least one AI provider key configured.
- **Brand Strategist AI response parsing:** If the AI returns malformed JSON, the service returns an empty array (no briefs created). No partial results.

### 19.2 Content Pipeline
- **Both competing drafts fail:** Content enters `generation_failed`. Human must retry (optionally with different `writer_ids`).
- **Image generation fails:** Draft proceeds without image. Image component stays in `failed` status for optional manual retry.
- **All component retries exhausted:** After 4 total attempts (initial + 3 retries), required component marks content as `generation_failed`.
- **Category deletion while content exists:** Blocked with error. Category can be deactivated but not deleted.
- **Category change on brief after generation:** Blocked with 409. Category is immutable once content items exist.

### 19.3 Scheduling & Publishing
- **No schedule configured:** Tenant operates in manual-only mode. No slot generation, no auto-scheduling.
- **Manual schedule outside publish window:** Slot is created anyway (manual override takes priority).
- **Publish webhook non-2xx:** 3 retries with exponential backoff (30s, 120s, 480s), then `publish_failed`.
- **Duplicate publish prevention:** Scheduler validates content item is in `scheduled` status before dispatching.
- **90-day auto-archive:** Status change only, no data deletion. `php yii archive/run` processes in batches of 100.

### 19.4 Authentication & Users
- **Multi-tenant login:** When user belongs to multiple tenants, login requires explicit `tenant_id` parameter (returns 400 without it).
- **Refresh token tenant context:** stored `tenant_id` and `role` restore context on refresh with no re-login required.
- **Admin-created user login:** 403 returned with `must_reset_password: true` hint until password is reset.
- **Owner cannot delete self:** Returns 409. Prevents orphaned tenants.
- **PlatformAdmin impersonation:** Always audit-logged. Impersonated token is scoped to the target tenant.

### 19.5 Data Integrity
- **Tenant isolation:** `TenantScopeTrait` on all tenant-owned models. Cross-tenant access returns 404 (not 403) to avoid leaking existence.
- **Encrypted fields:** AES-256 with app-level key. No per-tenant key hierarchy.
- **JSON columns:** `content_components.metadata` and `content_drafts.component_refs` use native MySQL JSON type for schema-level validation.
- **Concurrent slot filling:** no explicit locking; last-write-wins is acceptable for v2.0 since the consortium runs sequentially per tenant.

## 20. Resolved Decisions

**From MVP (carried forward):**
1. Content types: fixed global enum.
2. Archive policy: 90 days post-publish, no deletion.
3. AlsoAsked: single platform-level key.
4. Publish payload: per-tenant JSON template.
5. Rate limits: none.
6. Draft edit history: latest only, no snapshots.
7. Approver permissions: Editor, Admin, and Owner can all approve. No four-eyes rule.
8. Reviser auto-run: automatic after writer completion.
9. Competing drafts failure: proceed with winner; if both fail, block and require retry.
10. Brief approval: required before generation; supports monthly bulk pre-approval.
11. Brief -> content relationship: one brief -> one content item; refresh creates a new item with lineage.
12. Writer/reviser versioning: always latest production from Langfuse, no snapshotting.
13. Category deletion: blocked if referenced.
14. Single publish target per item.
15. No unpublish/takedown.

**From v1.0 (component pipeline):**
16. Queue backend: Dragonfly (Redis-compatible) via Docker.
17. Component-level generation: separate jobs per component with per-channel routing.
18. Per-writer model config: different AI provider + model per component type.
19. Image generation: Gemini 2.5; optional component, failure does not block text.
20. Content cadence: per-tenant scheduling with posts_per_week, publish_days, and a business-hours window.
21. Slot-based scheduling: pre-generated slots filled by the agent consortium or manual scheduling.
22. Agent consortium: automated brief selection with multi-criteria scoring.
23. AlsoAsked depth tracking: hierarchical question storage.
24. Auto-generation on approval: configurable per tenant.
25. Auto-scheduling: configurable per tenant.
26. Image storage: base64 in database; object storage migration planned.

**From v1.1 (Virtual Content Department):**
27. **Company profile:** optional rich tenant profile (products, audiences, brand voice, competitors, guidelines) wired into all AI prompts.
28. **AI Brand Strategist:** synchronous brief generation from topic + company context, creating briefs with `source = 'ai_strategist'`.
29. **Reviser types:** `brand_compliance`, `quality`, `seo`, `custom` with `revision_order` for execution sequence.
30. **Chained revision pipeline:** ordered chain of revisers (brand -> quality -> SEO) instead of a single default reviser. `REVISED -> REVISED` transition supports re-revision.
31. **Company profile in prompts:** full company context injected into writer prompts (`buildVariables`) and reviser prompts (`RunReviserJob`), plus fallback prompts.
32. **User management:** Owner can delete users, admin-created accounts with forced password reset, and 3 user creation modes.
33. **Tenant deletion:** PlatformAdmin soft-delete (sets status to inactive).
34. **Error envelope standardization:** all errors use `{success: false, error: {code, message, details}}`.
35. **Queue routing:** all dispatch sites go through `QueueRouter` for channel-specific routing.
36. **Container logging:** stderr output via `php://stderr` for container compatibility.
37. **Health check:** actual Redis PING for queue reachability and DB latency measurement.
